# Roadmap: v1.5 Lokale SEO-sider

## Overview

This milestone adds a scalable city landing page system targeting Norwegian local search queries. The architecture is config-driven static generation — a single `src/config/locations.ts` data file drives `getStaticPaths()`, all section components, footer linking, and structured data. The work proceeds in strict dependency order: infrastructure and the TypeScript interface must be locked before any city content is written, and all internal linking ships in the same deploy as the city pages. Phase 26 is a mandatory pause before any V2 expansion begins.

## Phases

- [ ] **Phase 24: Infrastruktur** - Lock `locations.ts` interface, URL structure, route skeleton, canonical pattern, and `tier`-gated `getStaticPaths()` — unblocks all city content downstream
- [ ] **Phase 25: Byinnhold og SEO** - Hand-craft 6–8 Tier 1 city entries with unique copy, city-specific FAQ, `Service` JSON-LD with `areaServed`, per-city metadata, footer linking, and `/kontakt` coverage mention — all ships in one deploy
- [ ] **Phase 26: Verifisering og V2-port** - Verify sitemap coverage after first deploy, validate structured data, document measurable V2 promotion criteria — mandatory gate before V2 begins

## Phase Details

### Phase 24: Infrastruktur
**Goal**: The `locations.ts` interface and route architecture are locked and verified building correctly, so city content can be authored against a stable foundation
**Depends on**: Nothing (first phase of milestone)
**Requirements**: INFRA-01, INFRA-02, INFRA-03, INFRA-04
**Success Criteria** (what must be TRUE):
  1. URL slug convention is documented in `.planning/` — ASCII-only slugs (`lillestrom`, `baerum`) with Norwegian display names in data; one entry in `locations.ts` confirms the pattern
  2. `locations.ts` exports a `City` TypeScript interface with all required fields (`tier`, `slug`, `name`, `intro`, `faq`, `nearbyAreas`, `industries`, `metaTitle`, `metaDescription`) — TypeScript strict mode accepts it at build time
  3. `src/pages/steder/[location].astro` with `getStaticPaths()` generates one static page per `locations.ts` entry filtered by active `tier`; `npm run build` passes with 1–2 stub entries
  4. Every generated city page contains a self-referencing `<link rel="canonical">` pointing to `https://nettup.no/steder/[slug]` — verified in rendered HTML; no `noindex` present
**Plans**: TBD

### Phase 25: Byinnhold og SEO
**Goal**: All 6–8 Tier 1 city pages are live with hand-crafted content that passes the doorway page test, complete JSON-LD, and full internal linking — everything ships in one deploy
**Depends on**: Phase 24
**Requirements**: CONTENT-01, CONTENT-02, SEO-01, SEO-02, LINK-01, LINK-02
**Success Criteria** (what must be TRUE):
  1. 6–8 city entries exist in `locations.ts` (Oslo, Drammen, Asker, Bærum, Lillestrøm, Sandvika, Ski, Moss) with intro paragraphs that are genuinely differentiated — no two pages share more than ~40% body content
  2. Every city page includes a city-specific FAQ section with 3–4 questions, at least 2 of which are genuinely local (e.g., industry mentions, geography, nearby areas) — not city-name substitutions into identical questions
  3. Every city page emits a `Service` JSON-LD block with `areaServed` referencing the global `LocalBusiness` entity via `"provider": {"@id": "https://nettup.no/#business"}` — no duplicate `LocalBusiness` declarations
  4. Every city page has a unique `<title>`, `<meta description>`, and `og:title` — no two cities share the same values
  5. Footer shows an "Omrader vi dekker" section listing all Tier 1 cities with links to `/steder/[slug]` — present in the same deploy as the city pages, not after
**Plans**: TBD

### Phase 26: Verifisering og V2-port
**Goal**: All V1 city pages are technically verified post-deploy, and measurable V2 promotion criteria are documented so V2 never starts without indexing confirmation
**Depends on**: Phase 25
**Requirements**: LINK-03, LINK-04
**Success Criteria** (what must be TRUE):
  1. After first deploy, a manual count of city URLs in `sitemap-index.xml` (or `sitemap-0.xml`) matches the number of active entries in `locations.ts` — discrepancy triggers the custom sitemap fallback endpoint before proceeding
  2. V2 promotion criteria are written in `.planning/` as a named checklist with concrete measurable thresholds (e.g., "All V1 pages show Indexed status in Search Console — not just Discovered", "≥3 V1 pages have organic impressions") — not vague qualitative language
