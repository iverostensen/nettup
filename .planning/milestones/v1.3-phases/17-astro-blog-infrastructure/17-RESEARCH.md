## Summary

Phase 17 implements Astro's Content Collections for a markdown blog at `/blogg` and `/blogg/[slug]`. The project uses Astro 5.16.6 (static output mode). The CONTEXT.md specifies using `src/content/config.ts` — the **legacy path**. This is intentional: Astro 5 moved the config to `src/content.config.ts` for the new Content Layer API, but the old path remains supported via the `legacy.collections: true` flag in `astro.config.mjs`. Using the legacy API (type: 'content') is the simpler choice for v1.3 since it avoids loaders and uses the familiar `slug`-based routing that matches the existing codebase patterns. Critically, `astro sync` must be run immediately after creating `src/content/config.ts` to generate `.astro/types.d.ts` — without this, TypeScript can't resolve `CollectionEntry<'blogg'>` and the build fails.

The structured data requirements (INFRA-03, INFRA-04) are handled via the existing `<slot name="head" />` pattern already used by service pages. BreadcrumbList is auto-generated by BaseLayout from `pageLabels` — the blog just needs `/blogg` and `/blogg/[slug]` entries added. BlogPosting JSON-LD and FAQPage JSON-LD are emitted per-article. The `relatedSlugs` undefined-filter requirement (INFRA-05) is a critical build-safety guard: when a slug referenced in frontmatter doesn't match any article, `getCollection()` returns `undefined` for that entry, causing a build crash if not filtered.

No new npm packages are needed. Content Collections and Zod (`z` from `astro/zod`) are built into Astro 5. All UI components needed (Card, Breadcrumbs, SectionHeader, Section) already exist. The only external changes are: adding `legacy: { collections: true }` to `astro.config.mjs`, adding "Blogg" to Footer nav, and adding blog page labels to BaseLayout's `pageLabels`.

**Primary recommendation:** Use the legacy Content Collections API (`src/content/config.ts` + `type: 'content'`) as explicitly specified, enable `legacy.collections: true` in `astro.config.mjs`, run `astro sync` before writing page components, and build everything on top of existing UI primitives.

### Pattern 1: Legacy Content Collections Config

**What:** Define the `blogg` collection using the Astro v2–v4 API at the legacy config path.
**When to use:** Explicitly decided by user — matches existing project conventions, simpler slug-based routing.

```typescript
// src/content/config.ts
// Source: https://docs.astro.build/en/guides/content-collections/ (legacy section)
import { defineCollection, z } from 'astro:content';

const blogg = defineCollection({
  type: 'content', // Markdown files, renders via render()
  schema: z.object({
    title: z.string(),           // Conversational — goes in <h1>
    seoTitle: z.string(),        // Keyword-first — goes in <title>
    category: z.string(),
    date: z.coerce.date(),
    readTime: z.number(),        // Minutes, integer
    description: z.string(),     // Meta description + og:description
    relatedSlugs: z.array(z.string()).optional(),
  }),
});

export const collections = { blogg };
```

**Required astro.config.mjs change:**
```javascript
// astro.config.mjs
export default defineConfig({
  // ... existing config ...
  legacy: {
    collections: true,  // Enable src/content/config.ts legacy path
  },
});
```

### Pattern 3: Dynamic Article Page with getStaticPaths

**What:** Generate static pages for each article, render markdown content.
**When to use:** `/blogg/[slug].astro`

```typescript
// src/pages/blogg/[slug].astro
// Source: https://docs.astro.build/en/guides/content-collections/
import { getCollection, render } from 'astro:content';
import type { CollectionEntry } from 'astro:content';

export async function getStaticPaths() {
  const articles = await getCollection('blogg');
  return articles.map((article) => ({
    params: { slug: article.slug },  // Legacy API uses .slug (not .id)
    props: { article },
  }));
}

interface Props {
  article: CollectionEntry<'blogg'>;
}

const { article } = Astro.props;
const { Content } = await render(article);
```

### Pattern 4: Related Articles with undefined-filter guard

**What:** Load related articles from slugs listed in frontmatter, filter out undefined.
**When to use:** RelatedArticles.astro component — INFRA-05

```typescript
// src/components/blog/RelatedArticles.astro
import { getCollection } from 'astro:content';

interface Props {
  relatedSlugs?: string[];
  currentSlug: string;
}

const { relatedSlugs = [], currentSlug } = Astro.props;

const allArticles = await getCollection('blogg');
const related = relatedSlugs
  .map(slug => allArticles.find(a => a.slug === slug))
  .filter((a): a is NonNullable<typeof a> => a !== undefined)  // CRITICAL: prevents build crash
  .slice(0, 3);
```

**What:** BaseLayout auto-generates BreadcrumbList JSON-LD from the `pageLabels` map.
**When to use:** Both `/blogg` and `/blogg/[slug]` — add entries to the map.

const pageLabels: Record<string, string> = {

### Anti-Patterns to Avoid

- **Using `post.id` instead of `post.slug` with legacy API:** The legacy Content Collections API uses `.slug` (filename without extension), not `.id`. The new Content Layer API uses `.id`. Mixing them causes 404s on dynamic routes.
- **Skipping `astro sync`:** Running `astro sync` after creating `src/content/config.ts` generates `.astro/types.d.ts`. Without it, TypeScript cannot resolve `CollectionEntry<'blogg'>` and the build fails with cryptic type errors.
- **No `legacy.collections: true` in astro.config.mjs:** Without this flag, Astro 5 ignores `src/content/config.ts` and tries to find `src/content.config.ts` instead. The collection won't exist and `getCollection('blogg')` returns `[]`.
- **Missing undefined filter in relatedSlugs:** If a `relatedSlugs` entry doesn't match any article slug, `.find()` returns `undefined`. Without the TypeScript type guard filter, a build crash occurs.
- **Passing `seoTitle` to `<h1>`:** The schema has two title fields for a reason. `seoTitle` (keyword-first, e.g. "Nettside pris 2026") must go to `<title>` only. The human-readable `title` goes to `<h1>`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown parsing | Custom MD parser | `render()` from `astro:content` | Handles remark/rehype plugins, returns typed `<Content />` component |
| Frontmatter validation | Manual type checking | `z.object()` Zod schema in `defineCollection` | Build-time validation with clear error messages |
| Slug generation | Custom filename→slug logic | Astro legacy API auto-derives `.slug` from filename | Consistent, handles special chars, matches routing |
| Reading time calculation | Character-count formula | Put `readTime` directly in frontmatter as a number | Simpler, manually accurate, no dependency needed |
| Related post lookup | Build a search index | `allArticles.find(a => a.slug === slug)` | Simple enough for ≤ 50 articles; no overhead |

**Key insight:** Astro Content Collections handle the hard parts (parsing, validation, type generation). The blog UI is just Astro components consuming typed data objects.

## Common Pitfalls

### Pitfall 1: Missing legacy.collections flag

**What goes wrong:** `getCollection('blogg')` returns an empty array at runtime. No build error — articles just silently don't appear.
**Why it happens:** Astro 5 defaults to the new Content Layer API. `src/content/config.ts` is only recognized if `legacy.collections: true` is set.
**How to avoid:** Add `legacy: { collections: true }` to `astro.config.mjs` before running `astro sync`.
**Warning signs:** Listing page renders with empty state despite `.md` files existing in `src/content/blogg/`.

### Pitfall 2: Skipping astro sync

**What goes wrong:** TypeScript errors on `CollectionEntry<'blogg'>` — type doesn't exist. `import { getCollection } from 'astro:content'` still works at runtime but the build may fail with TS errors depending on strictness.
**Why it happens:** `.astro/types.d.ts` is generated by `astro sync`, not by the dev server or build alone.
**How to avoid:** Run `npx astro sync` immediately after creating or modifying `src/content/config.ts`.
**Warning signs:** TypeScript shows "Cannot find module 'astro:content'" or "Type 'blogg' does not satisfy the constraint".

### Pitfall 3: relatedSlugs undefined entries crashing build

**What goes wrong:** Build crashes with `TypeError: Cannot read properties of undefined`.
**Why it happens:** `allArticles.find(a => a.slug === slug)` returns `undefined` if the slug doesn't match (typo, deleted article, etc.). Passing `undefined` to a component prop that expects `CollectionEntry` causes a crash.
**How to avoid:** Always filter: `.filter((a): a is NonNullable<typeof a> => a !== undefined)`.
**Warning signs:** Build succeeds locally (all slugs match) but fails in CI after an article is deleted or renamed.

### Pitfall 4: Using slug from file path vs. frontmatter

**What goes wrong:** If an article's filename is `hva-koster-en-nettside.md`, the auto-derived `.slug` is `hva-koster-en-nettside`. If `relatedSlugs` in another article uses a different string, the lookup fails silently.
**Why it happens:** The legacy API derives slug from the file path — no slug field in frontmatter.
**How to avoid:** Use exact filenames (without `.md`) in `relatedSlugs`. Standardize filenames to lowercase-kebab-case with no special characters.
**Warning signs:** RelatedArticles section is empty despite slugs appearing correct in frontmatter.

### Pitfall 5: BaseLayout BreadcrumbList shows raw slug

**What goes wrong:** The breadcrumb JSON-LD on article pages shows `"name": "hva-koster-en-nettside"` instead of the article title.
**Why it happens:** BaseLayout's `pageLabels` map doesn't have an entry for dynamic slug paths, so it falls back to the raw path segment.
**How to avoid:** Article pages use `Breadcrumbs.astro` directly (already decided) rather than relying on BaseLayout's JSON-LD breadcrumb for the article title. The BreadcrumbList JSON-LD for article pages should be emitted separately in the `head` slot with the correct article title — or add `/blogg` as a static entry and accept the slug as the last breadcrumb item.
**Warning signs:** Google Search Console shows breadcrumb structured data with URL-like strings instead of human-readable labels.

### Pitfall 6: og:type should be "article" for blog posts

**What goes wrong:** All article pages inherit `og:type="website"` from BaseLayout.
**Why it happens:** BaseLayout hardcodes `og:type="website"`.
**How to avoid:** Override via `<slot name="head" />` with `<meta property="og:type" content="article" />` on article pages. Also add `article:published_time` and `article:section` OG tags.
**Warning signs:** Facebook Debugger shows "website" type instead of "article" for blog post URLs.

|--------------|------------------|--------------|--------|

## Open Questions

1. **FAQPage JSON-LD source data**
   - What we know: INFRA-03 requires FAQPage JSON-LD on article pages. Seed articles are placeholder content.
   - What's unclear: Should the FAQ data come from a frontmatter `faq` field (structured) or be parsed from a `## Vanlige spørsmål` markdown section? Phase 18 (generation pipeline) adds FAQ sections to articles — PIPE-03 confirms this.
   - Recommendation: Add an optional `faq` frontmatter field (array of `{question, answer}` objects) to the schema. For seed articles, include 2-3 hardcoded FAQ items in frontmatter. This makes JSON-LD generation straightforward without parsing markdown.

2. **og:type override in BaseLayout**
   - What we know: BaseLayout hardcodes `og:type="website"`. Article pages should use `og:type="article"`.
   - What's unclear: Whether to modify BaseLayout to accept an `ogType` prop or emit the override via `<slot name="head" />`.
   - Recommendation: Emit `<meta property="og:type" content="article" />` via `<Fragment slot="head">` on article pages. The slot-injected tag will override the BaseLayout one in most parsers (technically invalid HTML, but practical). Alternatively, add an `ogType` prop to BaseLayout — a one-line change.
