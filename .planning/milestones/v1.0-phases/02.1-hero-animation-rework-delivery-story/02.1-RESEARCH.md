## Summary

The goal is to replace the static right-column stats card in `HeroIsland.tsx` with a cinematic animation that compresses Nettup's "2-week delivery" promise into ~3-4 seconds: a calendar fast-forwarding through 14 days, pieces of a website snapping into place, then a "Lansert!" badge appearing. This animation speaks directly to Norwegian SMB pain points rather than showing code (the old `LandingHeroAnimation.tsx` spoke to developers).

The installed Framer Motion is **v12.23.26** (confirmed from `node_modules`). All required APIs are present and exported: `useAnimate`, `animate`, `stagger`, `AnimatePresence`, `useReducedMotion`, `useMotionValue`, `motion`. No breaking changes were introduced in v12 for React APIs â€” the sequence animation syntax is stable since v11. No new dependencies are needed.

The recommended implementation approach is **`useAnimate` with an async sequence function** triggered via `useEffect`. This gives imperative timeline control over the multi-phase narrative (calendar â†’ build â†’ done), allows clean looping by calling the sequence function again after completion, and keeps all timing derived from `animation.ts` tokens.

**Primary recommendation:** Use `useAnimate` + async/await sequence (not variants, not keyframe arrays) to orchestrate the delivery story. Use a `useEffect` with cleanup for the loop. Apply `useReducedMotion` to render a static "done" state fallback.

### Pattern 1: useAnimate Async Sequence with Loop

**What:** `useAnimate` returns a `scope` ref and an `animate` function scoped to the component. Calling `animate(selector, keyframes, options)` returns a promise that resolves when the animation finishes. Awaiting in sequence creates a timeline.

**When to use:** Multi-phase narrative where phases must complete before the next begins.

**Sequence API format (verified from WebSearch + node_modules types):**

```typescript
// Source: motion.dev/docs/react-use-animate + confirmed from framer-motion dist/types
import { useAnimate, stagger, useReducedMotion } from 'framer-motion';
import { duration, springs, delay as delayTokens } from '@/lib/animation';

function HeroDeliveryAnimation() {
  const [scope, animate] = useAnimate();
  const shouldReduceMotion = useReducedMotion();

  useEffect(() => {
    if (shouldReduceMotion) return; // static fallback renders instead

    let cancelled = false;

    async function runSequence() {
      if (cancelled) return;

      // Phase 1: Calendar advances (days 1-14 flicker fast)
      // Drive via state + setInterval, or animate text content directly
      await animate('.day-counter', { opacity: [1, 0.4, 1] }, {
        duration: duration.normal,
        repeat: 6,  // 6 flickers = ~2s at 0.3s each
      });

      if (cancelled) return;

      // Phase 2: Website sections snap in with stagger
      await animate('.website-section', { opacity: [0, 1], y: [8, 0] }, {
        delay: stagger(0.1),
        duration: duration.fast,
      });

      if (cancelled) return;

      // Phase 3: "Lansert!" badge pops in
      await animate('.lansert-badge', { opacity: [0, 1], scale: [0.8, 1] }, {
        ...springs.pop,
        duration: duration.normal,
      });

      if (cancelled) return;

      // Rest state: pause before loop
      await new Promise(resolve => setTimeout(resolve, 2500));

      if (cancelled) return;

      // Reset to initial state and loop
      await animate('.day-counter', { opacity: 0 }, { duration: 0 });
      await animate('.website-section', { opacity: 0, y: 8 }, { duration: 0 });
      await animate('.lansert-badge', { opacity: 0, scale: 0.8 }, { duration: 0 });

      runSequence(); // loop
    }

    runSequence();

    return () => { cancelled = true; };
  }, [animate, shouldReduceMotion]);

  // ...
}
```

**Key insight:** The `cancelled` flag prevents state updates after unmount â€” critical for Astro view transitions where components can unmount mid-animation.

### Pattern 2: Day Counter via useState + setInterval

**What:** The "calendar advancing" phase is most readable as a JS counter that updates React state rather than trying to animate DOM text content directly with Framer Motion.

**Why:** Framer Motion's `animate()` function does not natively interpolate text content (it handles numeric CSS values). A day counter needs to increment through integer values 1-14.

```typescript
// Drive the calendar day counter
const [currentDay, setCurrentDay] = useState(1);

// In the sequence:
// Fire a fast interval for ~2 seconds to tick through days
// Then clear it before Phase 2
const dayTicker = setInterval(() => {
  setCurrentDay(d => Math.min(d + 1, 14));
}, 130); // 14 days * 130ms â‰ˆ 1.8s

await new Promise(resolve => setTimeout(resolve, 14 * 130 + 200));
clearInterval(dayTicker);
```

**Alternative (no state re-renders):** Use `useMotionValue` and `animate(motionValue, 14, { duration: 2 })` then `useTransform` to round to integer. This avoids re-renders but adds complexity.

### Pattern 6: Right Column Timing (Integration with Left Column)

**What:** Left column (`heroContainer`) fires immediately on mount. The right column delivery animation should start after the left column is mostly settled.

**Current right column delay:** `{ ...springs.gentle, delay: 0.35 }` on the `springPop` wrapper.

**Recommendation:** Keep the outer `motion.div` wrapper with `springPop` + `delay: 0.35` to animate the card container in. Then start the internal delivery sequence after the container animation settles (~0.35 + 0.5s spring settle = ~0.85s total). Add a `0.8` second initial delay before starting `runSequence()`.

```typescript
// In useEffect:

return () => { clearTimeout(timer); cancelled = true; };
```

**Important:** The `delay` used here is NOT from `animation.ts` (which uses seconds). The `setTimeout` takes milliseconds. Document this explicitly in the component.

### Anti-Patterns to Avoid

- **Animating text content directly with animate():** `animate('.day-counter', { innerHTML: 14 })` does not work. Use state.
- **Using `repeat: Infinity` on the whole sequence:** `repeat` on individual segments only repeats that segment. Loop the full sequence by calling `runSequence()` recursively.
- **Forgetting to cancel on unmount:** Astro view transitions unmount/remount islands. Without the `cancelled` flag, animations continue firing after unmount.
- **Hardcoding delays in milliseconds:** All `transition` options in `animate()` use seconds. The `setTimeout` for the rest pause uses milliseconds. Keep them clearly separated.
- **Putting `scope` ref on the wrong element:** `scope` must be a parent of ALL elements targeted by string selectors. If scope is on a child, selectors may not find siblings.
- **Using `motion.div` for elements inside the `useAnimate` scope:** Elements animated via `useAnimate` string selectors are plain DOM elements (divs, spans). They don't need to be `motion.*` components.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Animation sequencing | Custom Promise chain with requestAnimationFrame | `useAnimate` async/await | Built-in timing, cleanup, and interrupt handling |
| Stagger delays | Manual `delay: i * 0.1` in loop | `stagger(0.1)` from framer-motion | Single source of truth, respects `from` and `startDelay` options |
| Reduced motion detection | Custom `window.matchMedia` hook | `useReducedMotion()` from framer-motion | SSR-safe, updates on system preference change |
| Loop implementation | `setInterval` wrapping the whole sequence | Recursive `runSequence()` with `cancelled` guard | `setInterval` with `async` functions doesn't respect await boundaries |

**Key insight:** The complex part is the loop + cleanup, not the animations themselves. The `cancelled` flag + recursive call pattern is the correct approach.

## Common Pitfalls

### Pitfall 1: useAnimate Selector Scope

**What goes wrong:** `animate('.website-section', ...)` finds no elements because `scope` ref is not attached to a parent element, or the targeted elements render AFTER the `useEffect` runs.

**Why it happens:** The `scope` ref must be passed to a DOM element that is a parent of all animated children. Elements must exist in the DOM when `animate()` is called.

**How to avoid:** Attach `ref={scope}` to the outermost container div. Ensure the initial `opacity-0` state is in Tailwind (not controlled by Framer Motion `initial=`) so elements exist in DOM immediately.

**Warning signs:** Console warning "No elements found" or animation silently does nothing.

### Pitfall 2: Framer Motion animate() Transition Uses Seconds, setTimeout Uses Milliseconds

**What goes wrong:** Developer writes `{ duration: 150 }` expecting 150ms, gets 150 seconds.

**Why it happens:** Framer Motion transitions always use seconds. `animation.ts` exports values in seconds (e.g., `duration.fast = 0.15`). `setTimeout` takes milliseconds. The rest pause between loops uses `setTimeout(resolve, 2500)` â€” milliseconds.

**How to avoid:** Always use `animation.ts` tokens for Framer Motion transitions. Document the `setTimeout` values explicitly with `// ms` comment.

**Warning signs:** Animation takes an inexplicably long time.

### Pitfall 3: Loop Runs After Component Unmounts

**What goes wrong:** After navigating away (view transitions), the animation sequence continues to call `setCurrentDay` and `animate()` on unmounted elements, causing React errors.

**Why it happens:** `async` functions are not automatically cancelled when a component unmounts.

**How to avoid:** Set `cancelled = true` in the `useEffect` cleanup. Check `if (cancelled) return` before every `await` in the sequence. Use `clearTimeout` on any pending timers.

**Warning signs:** "Can't perform a React state update on an unmounted component" console errors.

### Pitfall 4: Spring Transition in animate() Sequence

**What goes wrong:** Passing `springs.snappy` (a spring object) to `animate()` inside `useAnimate` may not resolve duration correctly for sequencing â€” spring physics duration is open-ended.

**Why it happens:** Springs don't have a fixed `duration`, so awaiting a spring animation may not resolve at the expected time.

**How to avoid:** For the sequence phases, use `duration`-based transitions (tween). Reserve spring transitions for the outer container's `springPop` enter animation (which uses `motion.div` variants, not `useAnimate`).

**Warning signs:** Sequence pauses indefinitely or next phase fires immediately.

### Pitfall 5: Performance â€” Layout Thrashing with Day Counter

**What goes wrong:** Updating `currentDay` state 14 times in 1.8 seconds causes 14 React re-renders, each potentially causing layout recalculation.

**Why it happens:** `setInterval` + `setState` triggers a re-render on every tick.

**How to avoid:** Keep the component small. The day counter re-renders only the number text, not the whole card. Alternatively, use `useMotionValue` + `animate(motionValue, 14)` with `useTransform(mv, v => Math.round(v))` to avoid re-renders entirely.

**Warning signs:** Jank visible during day counter phase on lower-end devices.

## Animation Narrative Design

The planner needs to understand the visual structure to create tasks:

### Three-Phase Sequence

**Phase 1: Calendar (0.8s delay â†’ ~2s)**
- Display: Large day number (1-14) with "/ 14 dager" label
- Animation: State-driven counter ticking through 1-14 at ~130ms per day
- Background accent: subtle cyan dot or progress bar fills up
- Total duration: ~1.8s + small buffer

**Phase 2: Website Builds (immediately after Phase 1, ~0.5s)**
- Display: Browser chrome mockup with 4 abstract rectangles (nav/hero/cards/footer)
- Animation: rectangles snap in with `stagger(0.08)` from top to bottom
- Each rectangle: `opacity: [0, 1]`, `y: [6, 0]`, `duration: 0.15`

**Phase 3: Lansert! (immediately after Phase 2, ~0.3s)**
- Display: Cyan badge with checkmark + "Lansert!"
- Animation: `scale: [0.8, 1.05, 1]`, `opacity: [0, 1]` spring pop
- Stays visible during rest state

**Rest state (2.5s)**
- Everything stays visible
- Optional: subtle breathing pulse on "Lansert!" badge

**Loop reset (instant)**
- All elements snap back to initial opacity/position with `duration: 0`
- Day counter resets to 1
- `runSequence()` called again

### Reduced Motion Static Fallback

Show the "done" state (day 14, all website sections visible, Lansert! badge visible) as a plain static card. No transitions, no timers. Matches the existing HeroIsland.tsx reduced-motion pattern.

## Open Questions

1. **Day counter: state or MotionValue?**
   - What we know: Both work. State causes re-renders; MotionValue avoids them.
   - What's unclear: Whether 14 re-renders at 130ms intervals is measurable jank on mid-range devices.
   - Recommendation: Start with state (simpler). Switch to MotionValue if jank is observed during visual review.

2. **Calendar grid vs. large number**
   - What we know: A 2Ã—7 grid of cells filling in is more visually rich. A large number "14" is cleaner.
   - What's unclear: Which reads better at the card width (~320px).
   - Recommendation: Implement large number first; the grid can be a visual refinement.

3. **Loop count vs. infinite loop**
   - What we know: An infinite loop on a marketing page should not consume CPU indefinitely.
   - What's unclear: Whether Framer Motion's animations are GPU-only (they are for transform/opacity) or cause JS overhead per frame.
   - Recommendation: Loop indefinitely â€” transform/opacity animations are GPU-only in browsers. Add `will-change: opacity, transform` class if needed.

4. **Should "Lansert!" be the actual Norwegian word or a badge?**
   - What we know: "Lansert" = "Launched" in Norwegian. The constraint requires Norwegian text.
   - Recommendation: Use "âœ“ Lansert!" as a cyan badge. Clean and self-explanatory.
